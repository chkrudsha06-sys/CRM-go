# CRM 개발 프로젝트 지침서

이 문서는 CRM 프로그램을 Vercel과 GitHub 기반으로 개발할 때 따를 작업 기준입니다.  
프로젝트의 목표는 단순 고객관리 도구를 넘어서, 영업, 업무관리, 일정, 문서, 커뮤니케이션, 데이터 분석까지 한곳에서 처리할 수 있는 업무 플랫폼형 CRM을 만드는 것입니다.

## 1. 개발 협업 방식

사용자는 개발 초보자라는 전제를 기준으로 작업합니다.

기능을 요청받으면 코드를 생성하기 전에 아래 내용을 짧고 쉽게 설명합니다.

- 요청한 내용이 프로그램에서 어떤 방식으로 적용되는지
- 어떤 파일이나 화면에 반영되는지
- 실제로 반영되는 핵심 변경사항 요약

설명은 길게 하지 않고, 초보자가 이해할 수 있는 수준으로 간단하게 작성합니다.

## 2. 코드 생성 및 GitHub 반영 원칙

요청받은 기능을 구현한 뒤에는 가능한 한 바로 GitHub로 반영합니다.

기본 작업 흐름은 다음과 같습니다.

1. 요청 내용 확인
2. 기존 코드 구조 확인
3. 구현 방향 간단 설명
4. 코드 작성 또는 수정
5. 오류 검증
6. 변경사항 요약
7. GitHub 푸시

GitHub 푸시 전에는 반드시 변경된 파일과 주요 변경 내용을 확인합니다.  
민감정보, 테스트용 비밀번호, API 키, 불필요한 로그가 포함되지 않았는지 점검합니다.

## 3. 더 나은 구현 방식 제안

사용자가 특정 기능을 요청했더라도, 더 좋은 방식이 있다면 먼저 제안합니다.

제안할 때는 아래 기준을 따릅니다.

- 현재 요청을 그대로 구현하는 방법
- 더 안정적이거나 확장성 있는 방법
- 최신 웹 개발 기준에서 더 적합한 방법
- 유지보수와 사용자 경험 측면에서 더 좋은 방법

단, 제안은 너무 길게 하지 않고 선택하기 쉽게 설명합니다.  
명확하게 더 좋은 방식이 있다면 그 방식을 우선 적용하되, 적용 이유를 짧게 설명합니다.

## 4. 코드 검증 기준

오류 발생 빈도를 줄이기 위해 코드 검증은 한 가지 방식에만 의존하지 않습니다.

가능한 경우 아래 항목을 교차검증합니다.

- 타입 검사
- 린트 검사
- 빌드 검사
- 주요 화면 직접 확인
- 기능별 기본 동작 테스트
- 데이터 저장, 수정, 삭제 흐름 확인
- 예외 상황 확인

검증 결과는 작업 완료 시 짧게 공유합니다.  
검증을 실행하지 못한 경우에는 실행하지 못한 이유를 명확히 남깁니다.

## 5. CRM 제품 방향성

이 프로젝트는 고객관리만을 위한 단순 CRM이 아니라, 회사 업무를 통합 관리하는 시스템을 목표로 합니다.

기능을 설계할 때 아래 영역까지 확장 가능하도록 고려합니다.

- 고객 및 거래처 관리
- 영업 파이프라인 관리
- 상담 및 커뮤니케이션 기록
- 업무와 할 일 관리
- 일정과 미팅 관리
- 견적, 계약, 청구 관련 업무
- 파일 및 문서 관리
- 담당자별 권한 관리
- 알림과 리마인더
- 대시보드와 리포트
- 업무 자동화
- 외부 서비스 연동

새 기능을 추가할 때는 단일 화면에서 끝나는 기능이 아니라, 향후 다른 업무 기능과 연결될 수 있는 구조를 우선 고려합니다.

## 6. UI와 사용자 경험 기준

CRM은 반복적으로 사용하는 업무 도구이므로 화려함보다 명확함과 효율성을 우선합니다.

UI를 만들 때는 아래 기준을 따릅니다.

- 필요한 정보를 빠르게 찾을 수 있어야 합니다.
- 입력, 수정, 검색, 필터링이 편해야 합니다.
- 버튼과 메뉴의 의미가 명확해야 합니다.
- 모바일과 데스크톱에서 모두 사용 가능해야 합니다.
- 업무 흐름이 끊기지 않도록 화면 이동을 최소화합니다.
- 대시보드는 핵심 지표를 한눈에 볼 수 있어야 합니다.

## 7. 데이터 구조 설계 기준

CRM 데이터는 다른 기능과 연결될 가능성이 높으므로 처음부터 확장성을 고려합니다.

데이터 설계 시 아래 항목을 고려합니다.

- 고객, 회사, 담당자, 거래, 업무, 일정의 관계
- 생성일, 수정일, 담당자, 상태값 기록
- 삭제 대신 보관 또는 비활성 처리 가능성
- 검색과 필터링에 필요한 필드
- 권한별 접근 가능 범위
- 향후 리포트와 통계에 사용할 수 있는 구조

## 8. 보안과 권한 기준

CRM에는 고객정보와 업무정보가 포함되므로 보안을 중요하게 다룹니다.

기본 기준은 다음과 같습니다.

- 민감정보는 코드에 직접 작성하지 않습니다.
- 환경변수는 `.env` 파일이나 배포 환경 설정을 사용합니다.
- 사용자 권한에 따라 접근 가능한 정보를 분리합니다.
- 로그인, 인증, 권한 검사는 우회할 수 없도록 구현합니다.
- 고객 데이터 삭제와 수정은 신중하게 처리합니다.

## 9. 배포 기준

Vercel 배포를 기준으로 개발합니다.

배포 전에는 아래 항목을 확인합니다.

- 로컬 빌드 성공 여부
- 환경변수 설정 여부
- GitHub 최신 코드 반영 여부
- Vercel 배포 오류 여부
- 주요 페이지 접속 가능 여부
- 핵심 기능 동작 여부

## 10. 작업 완료 보고 형식

기능 구현 후에는 아래 형식으로 짧게 보고합니다.

```md
적용 내용:
- 무엇을 추가 또는 수정했는지

반영 위치:
- 변경된 주요 파일 또는 화면

검증:
- 실행한 검증 방법

GitHub:
- 푸시 완료 여부
```

보고는 초보자도 이해할 수 있도록 간단하고 명확하게 작성합니다.

## 11. 기본 개발 태도

이 프로젝트의 개발자는 단순히 요청받은 코드만 작성하지 않습니다.

항상 아래 관점으로 작업합니다.

- 사용자가 실제 업무에서 편하게 쓸 수 있는가
- 나중에 기능을 추가하기 쉬운 구조인가
- 오류가 발생했을 때 원인을 찾기 쉬운가
- 데이터가 안전하게 관리되는가
- CRM을 넘어 업무 플랫폼으로 확장 가능한가

이 기준을 바탕으로 모든 기능을 설계하고 구현합니다.
